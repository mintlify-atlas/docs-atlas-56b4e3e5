title: "Caching Strategies: Patterns and Redis Interview Q&A"
description: "Key interview questions on cache-aside, write-through, write-behind, cache stampede, Redis single-threaded design, distributed locking, and RAID storage."

Caching sits between your application and the underlying data store, absorbing read traffic and dramatically reducing latency. Choosing the wrong caching pattern — or misunderstanding how the cache handles concurrency and failures — can quietly degrade reliability under load. The questions below cover the write and eviction strategies that interviewers probe most often, along with Redis internals and distributed locking.

<AccordionGroup>
  <Accordion title="What is the Cache Aside (lazy loading) pattern and what are its trade-offs?">
    **Cache Aside** (also called lazy loading) is the most common read-through pattern. The application checks the cache first:

    - **Cache hit** — the value is returned directly from cache without touching the database.
    - **Cache miss** — the application fetches the value from the database, returns it to the caller, and simultaneously writes it into the cache for future requests.

    **Advantages:**

    - Only data that is actually requested ends up in the cache, so memory is used efficiently.
    - If the cache node fails or is unavailable, the application can fall back to the database transparently — the service keeps running.

    **Disadvantages:**

    - After a cold start or cache flush, many concurrent requests may all experience a cache miss simultaneously, causing a spike of database load (see the cache stampede question below).
    - Because the cache is only populated on a miss, there is always a window where the cache holds stale data if the underlying record changed between a write and the next miss.

    <Note>
      Cache Aside pairs naturally with TTL (time-to-live) expiry. Setting a sensible TTL limits how long stale data can linger without requiring explicit invalidation logic on every write path.
    </Note>
  </Accordion>

  <Accordion title="What write strategies can resolve cache inconsistency?">
    **Cache inconsistency** occurs when the cache and the primary database hold different values for the same key. Three patterns address this:

    **Write Through**

    Every write goes to both the cache and the database atomically. The cache always holds the current value, so reads are always fresh. The downside is that every mutation incurs two write operations, and data that is never re-read wastes that second write. Using a TTL limits unnecessary retention of write-only data.

    **Cache Invalidation**

    On a write, the cache entry for the affected key is deleted rather than updated. The next read will experience a miss, hit the database, and repopulate the cache with the fresh value. This avoids the double-write cost of Write Through and eliminates stale-read risk, but it does temporarily increase database load after a write.

    **Write Behind (Write Back)**

    Writes update the cache immediately and the database asynchronously, after a delay or in a background batch. This approach is excellent for write-heavy workloads because it decouples the user-facing response from slow disk I/O. The trade-off is that the cache and database are transiently inconsistent — if the cache node crashes before the async flush completes, writes in the buffer are lost.

    <Warning>
      Write Behind should only be adopted when the business can tolerate a small window of potential data loss. For financial transactions or audit logs, Write Through or immediate database writes are safer choices.
    </Warning>

    <Tip>
      In practice, many systems combine Cache Invalidation for write paths with Cache Aside for read paths. This keeps reads efficient while keeping the invalidation logic simple.
    </Tip>
  </Accordion>

  <Accordion title="What is a cache stampede (thundering herd) and how can it be prevented?">
    A **cache stampede** (also called a thundering herd) occurs when many requests all experience a cache miss at the same instant — typically right after a popular key expires or a cache node restarts. Every thread races to the database to reload the missing data, causing a sudden traffic spike that can overwhelm the primary store.

    Three mitigation strategies exist:

    **Locking**

    When a miss is detected, one thread acquires a lock on that cache key. Other threads wait. The lock holder fetches the data, populates the cache, then releases the lock — at which point waiting threads read from the now-populated cache. Locking is effective but introduces risk: deadlocks, lock-holder failures, and the overhead of the locking mechanism itself must all be managed carefully.

    **External recomputation**

    A dedicated background thread monitors cache keys and refreshes them proactively before they expire. This removes stampedes entirely because the key never becomes empty. The cost is that data which is never re-requested is refreshed unnecessarily, wasting compute and memory.

    **Probabilistic Early Recomputation (PER)**

    When a key is approaching its expiry time, each incoming request has a small, randomly calculated probability of triggering an early refresh. Because only a fraction of threads refresh at any given moment, the stampede is avoided without a hard lock. The Line Engineering team's `req-shield` library and Hwahae's blog post on the PER algorithm are real-world examples of this approach.

    <Note>
      For most services, PER offers the best balance between simplicity and effectiveness. Locking is a good fit when you need a strict guarantee that only one refresh ever runs concurrently for a given key.
    </Note>
  </Accordion>

  <Accordion title="Why is Redis single-threaded, and how does it still achieve high throughput?">
    Redis uses a **single-threaded event loop** for command execution. This design choice is deliberate:

    **Simplicity and correctness**

    Multi-threaded systems require locks, condition variables, and careful synchronization to prevent race conditions and deadlocks. A single-threaded model eliminates these hazards entirely. Every command is executed sequentially, so atomicity is guaranteed without any locking overhead.

    **Context-switching elimination**

    Because only one thread executes commands, the CPU never pays the cost of switching between threads. In-memory operations are so fast that the thread is rarely idle — context-switching overhead would represent a significant fraction of total work if multiple threads were involved.

    **Event-driven I/O**

    Redis handles network I/O through a non-blocking event loop (similar to Node.js). A single thread can multiplex thousands of client connections simultaneously, reading requests and writing responses asynchronously.

    **Redis 6.0 and threaded I/O**

    Starting in Redis 6.0, reading from and writing to client sockets is handled by multiple I/O threads. However, command execution itself remains single-threaded, so atomicity guarantees are preserved. The threaded I/O layer removes the network layer as a bottleneck on high-bandwidth deployments.

    ```
    Redis 6.0+ architecture:
    ┌──────────────────────────────────────────┐
    │  I/O threads (multi-threaded)            │
    │  ├── Read requests from client sockets   │
    │  └── Write responses to client sockets  │
    ├──────────────────────────────────────────┤
    │  Command executor (single-threaded)      │
    │  └── Processes commands sequentially     │
    └──────────────────────────────────────────┘
    ```
  </Accordion>

  <Accordion title="How does Redis implement distributed locking, and what is the RedLock algorithm?">
    Redis distributed locking uses the `SET` command with the `NX` (set-if-not-exists) option combined with an expiry (`EX`):

    ```bash
    SET lock:resource_id unique_token NX EX 30
    ```

    - If the key does not exist, the command succeeds and the caller holds the lock.
    - If the key already exists (another holder), the command returns `nil` and the caller must retry or fail.
    - When the work is complete, the lock holder deletes the key to release the lock.

    **The replication gap problem**

    In a master–replica setup, if the master crashes after granting a lock but before replicating that key to replicas, the promoted replica has no record of the lock. A second caller can then acquire the same lock — breaking mutual exclusion. This is the replication-lag vulnerability.

    **RedLock algorithm**

    To address this, Redis creator Salvatore Sanfilippo proposed **RedLock**, which uses multiple independent Redis nodes (no replication):

    1. Record the current timestamp.
    2. Attempt to acquire the lock on **all N nodes** (typically 5) using the same key and a unique token.
    3. The lock is considered acquired if the majority (⌊N/2⌋ + 1) of nodes granted it **and** the total time elapsed is less than the lock's TTL.
    4. If the majority is not reached, release any partial locks and retry.

    With 5 nodes, a caller must hold locks on at least 3 nodes simultaneously. A single node failure cannot grant the lock to two callers at the same time.

    <Warning>
      RedLock remains controversial in the distributed-systems community. Martin Kleppmann's critique points out that clock drift and GC pauses can still violate safety guarantees. For workloads requiring strict mutual exclusion, consider using a consensus system like etcd or ZooKeeper instead.
    </Warning>
  </Accordion>

  <Accordion title="What is RAID and how do the common RAID levels differ?">
    **RAID (Redundant Array of Independent Disks)** combines multiple physical disks into a single logical unit to improve performance, redundancy, or both.

    | Level | Technique | Performance | Fault tolerance |
    |-------|-----------|-------------|-----------------|
    | **RAID 0** | Striping — data is split across all disks | Highest read/write speed | None — one disk failure loses all data |
    | **RAID 1** | Mirroring — data is duplicated on every disk | Slower writes (must write to all copies) | Survives failure of all but one disk |
    | **RAID 4** | Striping with a dedicated parity disk | Bottleneck on parity disk | Single disk failure recoverable |
    | **RAID 5** | Striping with distributed parity | Balanced read performance | Single disk failure recoverable |
    | **RAID 6** | Striping with two independent parity blocks | Slightly slower writes than RAID 5 | Two simultaneous disk failures recoverable |

    **RAID 0** is ideal when raw speed matters and data loss is acceptable (scratch disks, temporary processing). **RAID 1** suits critical single-server data where simplicity of recovery matters most. **RAID 5** is the most common general-purpose choice — it balances usable capacity, performance, and fault tolerance. **RAID 6** adds a second parity layer for environments where double-drive failure must be survivable (large arrays where rebuild time is long).

    <Note>
      Managed cloud databases such as AWS RDS automatically stripe data across multiple EBS volumes internally, delivering RAID 0-like throughput without manual configuration. You rarely configure RAID explicitly when building on cloud infrastructure.
    </Note>
  </Accordion>
