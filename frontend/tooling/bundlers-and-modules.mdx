Build tooling spans a wide surface area in frontend interviews — from how browsers load scripts, to how package managers store dependencies, to how CI/CD pipelines automate delivery. This page collects the most frequently asked questions in this domain and provides structured, interview-ready answers grounded in real-world practice.

<AccordionGroup>
  <Accordion title="What is the difference between async and defer on a script tag?">
    Both attributes instruct the browser to download the script without blocking HTML parsing — but they differ in *when* the script executes.

    **`async`**
    - Downloads in parallel with HTML parsing.
    - Executes *immediately* when the download finishes, pausing HTML parsing for the duration of execution.
    - Multiple `async` scripts execute in download-completion order, not document order.

    **`defer`**
    - Downloads in parallel with HTML parsing.
    - Executes *after* the HTML document has been fully parsed, just before the `DOMContentLoaded` event fires.
    - Multiple `defer` scripts execute in document order.

    **Which to use when**

    | Script type | Attribute |
    |---|---|
    | Analytics, ads, independent widgets | `async` |
    | Main application logic, scripts with inter-dependencies | `defer` |
    | Inline scripts | Neither (they block by definition) |

    Using `defer` on your application bundle is the most common pattern because it preserves execution order and does not interrupt rendering.
  </Accordion>

  <Accordion title="What is the difference between CommonJS and ES Modules?">
    CommonJS (CJS) and ES Modules (ESM) are the two dominant module systems in the JavaScript ecosystem.

    **CommonJS**
    - Originated in Node.js; widely used on the server side.
    - Loads modules *synchronously* — execution halts until the required module is available.
    - Uses `require()` to import and `module.exports` to export.
    - Dynamic: `require()` calls can appear inside conditionals or functions.

    **ES Modules**
    - The official ECMAScript standard, introduced in ES2015.
    - Supported natively in browsers and in Node.js ≥ 12.
    - Loads modules *asynchronously*.
    - Uses `import` and `export` keywords.
    - *Statically analyzable*: imports must appear at the top level, which enables optimizations like tree shaking.

    ```javascript
    // CommonJS
    const lodash = require('lodash');
    module.exports = { myFunction };

    // ES Modules
    import { cloneDeep } from 'lodash';
    export { myFunction };
    ```

    <Tip>
      ESM is the modern default. Node.js projects increasingly adopt it for better browser/server compatibility and tree-shaking support. The ecosystem is still mid-migration, so you will encounter both in practice.
    </Tip>
  </Accordion>

  <Accordion title="Why do we need a bundler? What problems does it solve?">
    A bundler takes the many source files of your application and combines them into a smaller set of optimized files for delivery. The core motivations are:

    **1. Fewer network requests**

    A typical SPA might reference hundreds of JavaScript, CSS, and asset files. Each file requires a separate HTTP request, and the cumulative latency degrades load time. A bundler merges these into one or a few files.

    **2. Code optimization**

    - **Tree shaking** eliminates dead code (unused exports) before shipping.
    - **Dead code elimination** removes unreachable code paths.
    - **Minification** strips whitespace, comments, and shortens identifiers.

    **3. Transpilation and compatibility**

    Bundlers integrate with tools like Babel to downcompile modern JavaScript (optional chaining, async/await, ESM `import`) to ES5 for older browsers that do not support those features.

    Popular bundlers include **webpack**, **Vite** (which uses Rollup for production builds), **esbuild**, and **Parcel**. Each makes different trade-offs between configuration flexibility and out-of-the-box speed.
  </Accordion>

  <Accordion title="What is tree shaking, and what conditions are required for it to work?">
    Tree shaking is a dead-code elimination technique that removes unused exports from the final bundle. The name comes from the mental model of shaking a dependency tree so that loose (unused) code falls away.

    **Requirements**

    1. **ES Modules** — Tree shaking depends on static `import`/`export` analysis. Because ESM imports must appear at the top level, bundlers can determine at build time exactly which exports are consumed. CommonJS `require()` is dynamic, making this analysis unreliable.

    2. **Side-effect-free code** — If a module runs code with global side effects when imported (e.g., patching `Array.prototype`), the bundler cannot safely remove it. Marking a package as side-effect-free in `package.json` unlocks more aggressive elimination:

    ```json
    {
      "sideEffects": false
    }
    ```

    **Enabling tree shaking in webpack**

    Set `mode: 'production'` — that is sufficient for basic tree shaking. Adding `TerserPlugin` enables more aggressive optimizations.

    **Measuring the effect**

    Use `webpack-bundle-analyzer` to visualize bundle composition before and after, or compare raw bundle sizes in the build output.
  </Accordion>

  <Accordion title="What is the difference between dependency, devDependency, and peerDependency?">
    npm separates dependencies into three categories based on *when* and *how* they are needed.

    **`dependencies`**

    Packages required at runtime in production. If you use React to render your UI, `react` belongs here — it must be present for the application to function.

    **`devDependencies`**

    Packages needed only during development. Linters (`eslint`), test runners (`jest`), and bundlers (`webpack`) are development tools that should never ship to production. Placing them in `devDependencies` keeps your production install lean.

    **`peerDependencies`**

    Used primarily by library authors. A peer dependency signals that your library *requires* a particular version of another package to be present in the consumer's project, but you are not installing it yourself — the consumer must. This avoids shipping multiple copies of the same library.

    **Common mistakes and their consequences**

    - Putting a build tool in `dependencies` instead of `devDependencies` bloats the production artifact and can introduce security vulnerabilities unnecessarily.
    - Putting a runtime package in `devDependencies` causes production deployments to fail with a "module not found" error when `NODE_ENV=production` skips dev installs.
  </Accordion>

  <Accordion title="What is the difference between npm install and npm ci?">
    Both commands install project dependencies, but they behave differently and target different environments.

    | Behavior | `npm install` | `npm ci` |
    |---|---|---|
    | Version resolution | Resolves within the range in `package.json` | Reads exact versions from `package-lock.json` |
    | Modifies lockfile | Yes, if versions change | Never |
    | Cleans `node_modules` first | No | Yes, always |
    | Speed | Slower (must resolve) | Faster (no resolution needed) |
    | Intended environment | Local development | CI/CD pipelines |

    **Key takeaway:** `npm ci` guarantees a reproducible, deterministic install by strictly following the lockfile. This is critical in CI because it ensures every build runs against the same dependency tree, regardless of what has been published to the npm registry since the lockfile was written.

    Running `npm ci` locally is valid but inefficient — it deletes and reinstalls `node_modules` every time, which is unnecessary for day-to-day development.
  </Accordion>

  <Accordion title="What is the difference between pnpm and Yarn Berry?">
    Both are modern alternatives to classic npm and Yarn v1, solving the same core problems (slow installs, wasted disk space, phantom dependencies) via different mechanisms.

    **pnpm**

    pnpm maintains a single global *content-addressable store* on disk. Every project's `node_modules` contains only hard links and symlinks pointing into that store — no physical copies. Benefits:

    - Packages shared across projects take no extra disk space.
    - Installation is fast (link creation vs. file copying).
    - Strict dependency isolation by default — no *phantom dependencies* (packages accessible but not declared in your own `package.json`).
    - First-class monorepo support via workspaces.

    **Yarn Berry (Yarn v2+)**

    Yarn Berry replaces `node_modules` with the *Plug'n'Play (PnP)* mechanism:

    - Dependencies are stored as zip archives in `.yarn/cache/`.
    - A `.pnp.cjs` file maps every package name to its zip location — no filesystem traversal required, making module resolution faster and more reliable.
    - Because the archives are small, they can be committed to Git — this is called **Zero Install**. CI pipelines can skip the install step entirely:

    ```yaml
    jobs:
      build:
        steps:
          - uses: actions/checkout@v2
          - uses: actions/setup-node@v2
          - run: yarn test  # no yarn install needed
    ```

    Both pnpm and Yarn Berry eliminate phantom dependencies and support workspaces. The main practical difference is the storage mechanism (symlinks vs. PnP) and whether you want Zero Install.
  </Accordion>

  <Accordion title="What is the difference between Yarn (v1) and Yarn Berry?">
    Yarn v1 (often just called "Yarn") follows the same `node_modules`-based model as npm, with performance improvements but no fundamental architectural change.

    Yarn Berry (v2+) is a complete rewrite with a different philosophy:

    - **No `node_modules`** — replaced by `.pnp.cjs` and zip-archived packages in `.yarn/cache/`.
    - **Zero Install** — compressed archives are small enough to commit to version control, letting collaborators and CI environments skip installation entirely.
    - **Deterministic resolution** — the `.pnp.cjs` file contains the full dependency graph with exact locations, eliminating the ambiguity of `node_modules` traversal.

    The trade-off is ecosystem compatibility: some tools that hardcode assumptions about `node_modules` need extra configuration (or the `node_modules` linker mode) to work with Yarn Berry's PnP.
  </Accordion>

  <Accordion title="What is a monorepo, and what are its trade-offs?">
    A monorepo stores multiple projects or packages in a single version-controlled repository, as opposed to maintaining a separate repo per project (a *polyrepo* approach).

    **Common structure**

    ```
    /
    ├── apps/
    │   ├── web/
    │   └── mobile/
    ├── packages/
    │   ├── ui/        ← shared component library
    │   └── utils/     ← shared utilities
    └── package.json   ← workspace root
    ```

    Workspace features in npm, Yarn, and pnpm hoist shared dependencies to the root, preventing duplication.

    **Advantages**

    - **Code sharing** — shared utilities and components live alongside the code that consumes them; a single change propagates immediately.
    - **Consistent tooling** — one ESLint config, one TypeScript config, one CI definition.
    - **Atomic commits** — a change that touches multiple packages can land in a single commit with a single PR.
    - **Easier cross-package refactoring**.

    **Disadvantages**

    - **Build and test times grow** with the codebase. Tools like **Nx** and **Turborepo** mitigate this with task caching and affected-only execution.
    - **Access control is coarse** — everyone with repo access can read all projects.
    - **CI/CD complexity increases** — pipelines must understand which packages are affected by a given change.

    Monorepos work best for teams that own multiple related packages (e.g., a component library, several apps, and shared utilities). For a single, self-contained application, the overhead rarely pays off.
  </Accordion>

  <Accordion title="What are the main Git branching strategies?">
    Three strategies dominate frontend teams:

    **Git Flow**

    Uses long-lived branches: `main`, `develop`, `feature/*`, `release/*`, and `hotfix/*`. Features merge into `develop`; `develop` merges into `release` for QA; `release` merges into `main` for production. Hotfixes branch off `main` directly and merge back into both `main` and `develop`.

    Best for: projects with scheduled releases and strict QA gates (e.g., finance, healthcare).

    Drawback: high branching complexity; merge conflicts accumulate.

    **GitHub Flow**

    One permanent branch (`main`). Every change starts as a feature branch, goes through code review via pull request, and merges directly to `main` for immediate deployment.

    Best for: teams deploying frequently (multiple times per day).

    Drawback: no dedicated QA buffer before production.

    **Trunk-Based Development**

    Developers commit directly to `main` (or merge very short-lived branches within a day or two). Relies heavily on feature flags and automated testing to keep `main` always releasable.

    Best for: high-velocity teams with mature CI/CD and feature flag infrastructure.

    Drawback: requires strong automated safety nets; risky without them.

    <Tip>
      There is no universal winner. Choose based on your release cadence, team size, and quality assurance requirements.
    </Tip>
  </Accordion>

  <Accordion title="What is CI/CD and why does it matter?">
    CI/CD stands for Continuous Integration and Continuous Delivery/Deployment. It automates the path from a developer's code change to a running production service.

    **Continuous Integration (CI)**

    Every code change triggers an automated pipeline that builds the project and runs the test suite. Problems are caught before code reaches the main branch, not after. In a team environment this prevents one developer's changes from silently breaking another's work.

    **Continuous Delivery vs. Continuous Deployment**

    - *Continuous Delivery* — automation prepares a release artifact that is ready to deploy, but a human manually approves the final push to production.
    - *Continuous Deployment* — the entire pipeline is automated end-to-end; every passing build ships to production automatically.

    **A typical pipeline**

    1. Developer opens a pull request.
    2. CI runs linting, type checking, unit tests, and build.
    3. If everything passes, the PR can be merged.
    4. Merging to `main` triggers a deployment to staging, then production.

    Common CI/CD platforms in frontend work: **GitHub Actions**, **GitLab CI**, **Jenkins**, **CircleCI**.
  </Accordion>

  <Accordion title="What is Node.js and what makes it well-suited for server-side JavaScript?">
    Node.js is a JavaScript runtime built on Chrome's V8 engine. It lets JavaScript execute outside the browser, making it possible to build servers, CLIs, and tooling with the same language used on the frontend.

    Three architectural properties define Node.js:

    **1. Single-threaded execution**

    Node runs JavaScript on a single thread. Unlike traditional server models that spawn a new OS thread per request, Node handles everything on one thread, which reduces overhead and memory usage for I/O-heavy workloads.

    **2. Event-driven architecture**

    Node uses an event loop. When an async operation (database query, file read, network request) completes, its callback is placed in a task queue and picked up by the event loop when the main thread is free. This means the thread is never idle waiting for I/O.

    **3. Non-blocking I/O**

    I/O calls (reading a file, querying a database) return immediately and notify Node when they finish. The thread continues processing other work in the meantime, enabling high throughput with low resource usage.

    **When Node.js is not the right tool**

    Node's single thread becomes a bottleneck for CPU-intensive work — complex image processing, machine learning inference, or heavy numerical computation. For those tasks, a multi-threaded runtime or worker threads are better suited.
  </Accordion>

  <Accordion title="What is Storybook and what problems does it solve?">
    Storybook is a development environment for building and documenting UI components in isolation. A *story* defines a specific state of a component — for example, a `Button` in its default, disabled, and loading states each become separate stories.

    **Core benefits**

    - **Isolated development** — you build and verify a component without running the entire application, speeding up iteration.
    - **Visual documentation** — designers, PMs, and QA can browse all component states in a living catalogue without reading code.
    - **Cross-team collaboration** — the visual catalogue reduces communication overhead between developers and non-developers.
    - **Design system support** — Storybook is the standard tool for publishing a shared component library.

    **Trade-offs**

    - Maintaining story files adds ongoing overhead, especially for large component libraries.
    - Components that depend heavily on server data require mock setup.
    - Some library integrations require extra configuration or compatibility workarounds.

    Storybook pays off most clearly on teams building a design system or a component library consumed by multiple applications.
  </Accordion>

  <Accordion title="What is JSON Schema and how is it used in frontend development?">
    JSON Schema is a specification for describing the structure, types, and constraints of JSON data. Think of it as a contract that defines what valid data looks like.

    **Example schema for a user registration payload**

    ```json
    {
      "type": "object",
      "properties": {
        "username": { "type": "string", "minLength": 3 },
        "email":    { "type": "string", "format": "email" },
        "password": { "type": "string", "minLength": 6 }
      },
      "required": ["username", "email", "password"],
      "additionalProperties": false
    }
    ```

    **Frontend use cases**

    1. **API response validation** — validate incoming data against a schema before trusting it, catching backend inconsistencies early.

    2. **TypeScript type generation** — tools like `json-schema-to-typescript` automatically generate TypeScript interfaces from a schema, keeping types and API contracts in sync:

       ```bash
       npx json-schema-to-typescript schema.json -o types.ts
       ```

    3. **Config file validation** — `.eslintrc`, `tsconfig.json`, and `prettierrc` are all described by JSON Schemas published to the JSON Schema Store. Editors like VS Code use these schemas to provide autocompletion and inline error reporting when you edit config files.
  </Accordion>

  <Accordion title="How do you approach debugging a complex frontend problem?">
    Effective debugging follows a structured process rather than trial-and-error guessing.

    **1. Read the error carefully**

    The error message and stack trace usually contain the most direct information about what went wrong and where. Skimming error messages is one of the most common time-wasting habits.

    **2. Isolate the problem**

    Reproduce the bug in the smallest possible environment. If a complex component is misbehaving, extract the failing part and test it with minimal data. Reducing scope eliminates confounding variables.

    **3. Compare expected vs. actual behavior**

    Sketch the correct execution flow. Then trace the actual flow under the bug conditions. The point where they diverge is typically the root cause.

    **4. Look for known solutions**

    Check the official documentation, GitHub Issues of the relevant library, and reputable articles before reaching for a search engine or AI tool. This order matters — official sources are more reliable and comprehensive.

    **5. Use the right tools**

    - Browser DevTools (Network, Console, Sources, Performance panels)
    - IDE debugger with breakpoints
    - React DevTools / Vue Devtools for component-level inspection

    After resolving the issue, document what you found and how you fixed it. The same bug pattern often recurs, and your notes become a resource for the team.
  </Accordion>
